# 3. Методика испытаний и программа

## 3.1 Цель и задачи испытаний

Целью испытаний является убеждение в корректности функционирования разработанной утилиты `${APP_NAME}`, оценка функциональной избыточности и соответствия требованиям технического задания.

Испытания проводятся для подтверждения того, что система:

- корректно обрабатывает пользовательские CLI-команды;

- обеспечивает взаимодействие с базой данных;

- выполняет работу с шаблонами документов;

- формирует итоговые файлы без ошибок;

- устойчиво реагирует на некорректные входные данные и выводит информацию об ошибке.

Задачи испытаний включают проверку всех команд, указанных в справке `--help`, а именно:

- CRUD-операций базы данных;

- работу шаблонного механизма;

- генерацию документов.

---

## 3.2 Объект испытаний

Объектом испытаний является исполняемый файл разработанной CLI-утилиты `${APP_NAME}`, полученный в результате сборки исходного кода проекта на языке C++.

Испытания проводятся над скомпилированным приложением, однако проверяемый функционал реализован следующими компонентами исходного кода:

- `CommandRouter` — подсистема маршрутизации и обработки CLI-команд;

- обработчики команд (`DbCommand`, `DataCommand`, `TemplateCommand`, `GenerateCommand`, `HelpCommand`, `VersionCommand`);

- `Database` — подсистема хранения и обработки данных.

---

## 3.3 Условия проведения испытаний

Испытания доступны к проведению в следующих программно-аппаратных средах:

- операционные системы: Windows, Linux и macOS;

- интерфейс взаимодействия: терминал (командная строка);

- структура проекта: локальная файловая система;

- запуск: путём выполнения собранного исполняемого файла с передачей параметров на целевой платформе.

Пример запуска:

```bash
./${APP_NAME} --help
```

Для повышения удобства эксплуатации допускается добавление каталога с исполняемым файлом в системную переменную окружения PATH. В этом случае запуск приложения может осуществляться из любого каталога без указания полного пути к исполняемому файлу.

---

## 3.4 Методика проведения испытаний

Методика испытаний основана на тестировании исключительно функционала, таким образом проверяется соответствие входных CLI-команд и ожидаемый результат без анализа внутренней реализации.

Испытания доступны к проведению по следующим направлениям:

1. **Проверка справочной системы**

Команда:

```bash
${APP_NAME} --help
```

Проверяется:

- наличие всех команд;

- корректность описания;

- соответствие фактическому функционалу `CommandRouter`.

---

2. **Проверка управления базой данных**

Команды:

```bash
db --init
db --init --templates
db --init --template students
db --list
db --schema-show <table_name>
db --drop <table_name>
db --drop --all
```

Проверяется:

- создание таблиц;

- удаление таблиц;

- отображение структуры;

- корректная обработка SQL-операций через `Database.cpp`.

---

3. **Проверка работы с данными**

Команды:

```bash
data --show
data --show students
```

Проверяется:

- корректный вывод записей;

- обработка пустых таблиц;

- устойчивость к неверным именам таблиц.

---

4. **Проверка шаблонов**

Команды:

```bash
template --list
template --add file.docx -as letter
template --show letter
template --remove letter
```

Проверяется:

- регистрация шаблонов;

- корректность загрузки файлов из `resources/templates`;

- работа с именами шаблонов.

---

5. **Проверка генерации документов**

Команда:

```bash
generate --source students --template letter --output output/
```

Проверяется:

- извлечение данных из БД;

- подстановка значений в шаблон;

- формирование DOCX-файла;

- сохранение результата в директорию `output/`.

Фактически проверяется ключевой сценарий системы, аналогичный работе документ-генераторов уровня Pandoc, но с привязкой к базе данных.

---

## 3.5 Тестовые сценарии

**Сценарий 1: Полный цикл генерации документа**

1. Инициализация базы:

```bash
db --init --template students
```

2. Просмотр данных:

```bash
data --show students
```

3. Выбор шаблона:

```bash
template --show wordfile.docx
```

4. Генерация документа:

```bash
generate --source students --template wordfile.docx --output output/
```

Ожидаемый результат:

- появление файла `output/project.docx`;

- корректная подстановка данных;

- сохранение форматирования шаблона.

---

**Сценарий 2: Проверка устойчивости к ошибкам**

Команды:

```bash
db --drop unknown_table
generate --source wrong --template none
template --remove missing
```

Ожидаемый результат:

- вывод сообщений об ошибке;

- отсутствие падения программы;

- сохранение работоспособности CLI.

---

## 3.6 Критерии успешности испытаний

Система считается успешно прошедшей испытания при выполнении следующих условий:

- все команды из `--help` корректно выполняются;

- приложение успешно запускается и выполняет заявленные функции на операционных системах Windows, Linux и macOS;

- отсутствуют критические ошибки выполнения;

- данные корректно извлекаются из БД (`Database.cpp`);

- шаблоны обрабатываются без потери структуры;

- документ успешно генерируется в `output/`;

- система устойчиво работает при ошибочном вводе.

---

## 3.7 Вывод по испытаниям

Проведённые испытания показали, что консольная утилита `${APP_NAME}` функционирует корректно в рамках заявленной архитектуры и соответствует требованию кроссплатформенности, предусматривающему возможность её использования на распространённых операционных системах Windows, Linux и macOS.

Основной сценарий работы — генерация документов на основе базы данных и шаблонов — выполняется без ошибок, что подтверждает работоспособность подсистемы обработки команд (`CommandRouter`), обработчиков команд и подсистемы хранения данных (`Database`).

Система демонстрирует устойчивость к некорректным входным данным и соответствует концепции специализированного CLI-инструмента для автоматизированной генерации документов.
